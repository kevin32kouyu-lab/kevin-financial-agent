# Financial Agent 10周成熟化路线图（5阶段）

更新时间：2026-04-18

## 背景与目标

这份路线图用于把当前项目从“可演示原型”升级到“成熟 Agent 产品形态”。  
核心方向是：在不破坏现有可用性的前提下，逐步补齐输入理解、记忆、规划、工具、知识、校验和治理能力。  

本路线锁定策略：

- 前台聚焦 Terminal，不扩展开发者展示页
- 后端接口尽量保持兼容，减少迁移成本
- 每阶段都给出可验收结果，避免只做概念

## 里程碑时间表（10周）

## 阶段1（第1-2周）：产品界面与交互完成度

目标：让 Terminal 达到“可直接给非技术用户使用”的体验。  

交付物：

- 统一视觉系统（颜色、字体、卡片层级）
- 统一提示体系（成功、失败、风险、新手引导）
- 关键流程动效（加载、切换、反馈）
- 页面文案去调试化，强调用户语言

验收：

- 用户 3 分钟内可独立完成一次研究和回测
- 页面信息层级清晰，无明显学习门槛

## 阶段2（第3-4周）：输入理解 + 记忆系统

目标：让 Agent 真正理解用户并“记得住”。  

交付物：

- 输入理解模块：把自然语言转成结构化投资目标
- 记忆系统：长期偏好（风险、期限、收益目标）与近期上下文记忆
- 澄清机制：信息不足时给出可读追问，而不是直接失败

验收：

- 同类问题重复提问时，系统不再重复问相同条件
- 任务意图可视化展示为“用户画像 + 本次目标”

## 阶段3（第5-6周）：规划与执行中台

目标：让流程可拆解、可追踪、可恢复。  

交付物：

- 规划层：自动拆解任务步骤并显示进度
- 工具层：统一工具注册与调用入口（MCP/Skills 兼容）
- 执行层：统一重试、取消、降级、超时策略

验收：

- 失败任务可恢复，不影响整单流程
- 同一流程在多次运行下结果稳定

## 阶段4（第7-8周）：知识层与校验层

目标：让报告“有依据、可追溯、能自检”。  

交付物：

- 知识层（RAG）：建立研究资料检索与引用能力
- 校验层：结论与数据的一致性检查
- 结果反思：关键结论给出可信度和证据来源

验收：

- 关键结论可追溯到来源与时间
- 出现数据冲突时会自动降级并明确提示

## 阶段5（第9-10周）：安全治理与发布

目标：达到对外展示和持续迭代的发布标准。  

交付物：

- 密钥与权限治理（防泄露、防误用）
- 审计日志（谁在何时触发什么任务、产生什么结论）
- 发布前稳定性检查（回归、性能、容错）

验收：

- 仓库和部署环境无明文密钥
- 核心用户流程可稳定复现

## 每阶段总交付清单（面向展示）

- 一个持续可演示的 Terminal 页面
- 一套可解释的研究报告与回测结果
- 一份可追溯的证据与风险说明
- 一套可对外讲清楚的 Agent 架构叙事

## 风险与应对

- 风险：外部免费数据源存在限流和不稳定。  
  应对：主源 + 备用源 + 缓存 + 可读降级提示。  

- 风险：阶段并行开发可能造成体验与稳定性冲突。  
  应对：每阶段先保主流程，再做增强项。  

- 风险：需求变化快，容易造成路线漂移。  
  应对：每周固定里程碑复盘，未完成项下周顺延。  

## 验证流程（长期路线每周复用）

1. 实时研究：输入投资目标，生成正式报告。  
2. 历史研究：指定 as_of_date，生成历史视角结论。  
3. 回测对照：查看组合 vs SPY，并解释跑赢/跑输原因。  
4. 导出报告：导出 HTML/PDF，用于结果留存和复核。
