## 近期关键决定与原因
- 火山 Key 修复先做“本地校验 + 清晰报错”，不硬猜 Key 格式，是因为没有真实密钥时不可能真正连通火山，但可以先把错误从误导性的 401 收敛成明确配置问题。
- 在拿到真实 Key 后继续做一次最小接口验证，而不是直接假设“写进 `.env` 就算修好”，是为了先把认证问题和后续的报告链路问题分开。
- 这轮实验采用“小样本功能验证”而不是正式 benchmark，是为了先证明系统能跑通、能解释、能展示。
- 不直接改用户 PPT，而是输出两张 16:9 图片，是为了让用户能更快拖进现有页面，不受原文件结构影响。
- 结果页保留“显式点名股票跟随不稳定”和“长文报告回退到结构化输出”这两个真实发现，是为了答辩时更可信，不夸大系统能力。
- 先完整读懂现有项目，再进入下一轮修改或排查，是为了减少对现有链路、回退逻辑和前台展示层的误判。
- 在 Railway 还没有可直接登录部署条件时，先用 localtunnel 生成一个临时公网网址，是为了先满足“马上能给别人访问”的展示需求。
- 临时公网链接仍然建立在本机服务之上，因此机器睡眠、关机或进程退出后，链接会失效；长期展示仍然以 Railway 为正式路线。
- 公开展示默认优先选 Railway，而不是临时隧道：因为当前项目已经是 Docker 单服务结构，接 GitHub 后最省事，也更稳定。
- 不把环境变量写进仓库，而是放 Railway 面板：这样更安全，也符合公开展示的实际使用方式。
- 增加 `/healthz` 而不是让平台直接探首页：这样能更稳定判断服务是否活着，也避免首页资源加载影响探活结果。
- 容器启动改用 `python main.py` 读取运行时配置，而不是把端口写死在 Docker 命令里：这样 Railway 分配的 `PORT` 会自动生效。
- 轻量记忆继续保留浏览器本地副本，但同步写入后端 SQLite，是为了让“最近偏好”不再只依赖单浏览器状态，后续接持久化记忆也更顺。
- 历史页不直接消费整份 run 原始结果，而是单独用 `audit-summary` 接口拿简版摘要，是为了让前台更稳定，也避免历史页被原始结构变化频繁打断。
- `needs_clarification` 状态下继续研究采用“沿用原问题 + 追加一句补充信息”的方式，是为了让用户感觉像自然续写，而不是重新填一张新表单。
- 导出内容补上“可信度等级 / 覆盖提醒 / 本次沿用的偏好”，是为了继续缩小“页面能看懂、导出看不全”的落差。
- 结论页新增“可信度概览”而不是只把这些信息塞进正式报告里，是为了让用户在第一屏就看到：系统理解了什么、为什么这样判断、哪里要保守。
- 澄清问题改成一句短追问，而不是多句展开，是为了让补充信息这一步更像对话，而不是像表单报错。
- 标准演示问题固定为 3 条，并给历史模式绑定固定日期，是为了让演示脚本可重复、可预期，不再受默认日期漂移影响。
- 导出内容继续向页面对齐，优先补“原始问题 + 研究模式 + 候选依据/谨慎提示”，是为了减少“页面看得到、导出看不到”的落差。
- 轻量记忆采用“前端本地保存 + 后端只补缺不覆盖”的方式，是为了先快速得到连续上下文体验，又避免把旧偏好强行套到新问题上。
- “最小可用 RAG”先不做重型知识库，而是直接从现有结构化研究结果里提取依据摘要，是为了先把演示稳定性和可解释性做好。
- 结论页新增“用户画像 / 依据 / 校验 / 安全摘要”，是为了让非技术用户第一眼就能判断：系统理解我了吗、这份结论靠谱吗、哪里要谨慎。
- 导出逻辑改为优先沿用页面同一套摘要数据，而不是重新拼另一套文案，是为了避免“页面一套、导出一套”的断层。
- 首页定位正式锁定为“品牌入口”，不是单纯的跳转页，这样更适合演示和第一次接触的用户。
- Terminal 现阶段改为“三页化前台”，而不是把结论、回测、历史继续堆在一页，是为了让用户更容易理解和演示。
- 结论页首屏用“摘要卡片”替代“巨大结论海报”，是为了同时看清结论、风险、动作和原始问题。
- 当前 active run 通过 URL 参数保留，是为了在结论页、回测页、历史页之间切换时不丢上下文。
- 进度区改成“阶段名称 + 高对比进度条 + 状态说明”的三层结构，是为了在深色背景下仍然清楚可读。
- 终端主标题与首页主标题采用“品牌感更强的展示字 + 更稳的正文无衬线”组合，是为了让第一眼更高级、阅读也更清楚。
- 逐票研究卡片从长段落改为分组阅读，是为了避免用户在候选池里快速失去焦点。
- 本轮前端升级不改后端接口，只重组前端层，是为了在视觉和交互大改的同时不破坏研究与回测主链路。
- 动效采用“可关闭 + 自动降级”策略，是为了在保证质感的同时兼顾低性能设备和可访问性。

- 终端页继续不改后端接口，只做展示层升级，是为了降低研究与回测链路回归风险。

- 首页采用“价值说明 + 三步引导 + 快速进入终端”，是为了让非技术用户第一眼就能理解项目用途。
- Terminal 继续保留既有研究/回测链路，只做展示重构，是为了在演示体验提升的同时避免业务回退。
- 历史模式采用“严格 as_of_date”原则：拿不到历史数据就显式降级，避免未来数据污染历史结论。

- 保留 `/terminal`（用户）和 `/debug`（开发者）双界面，兼顾产品体验和可观测性。
- 取消运行中任务采用“状态立即取消 + 最佳努力停止后台执行”，优先保证用户侧反馈及时且稳定。

- 报告语言继续按“用户提问语言”锁定，界面语言只控制壳层 UI，避免正文被误切换。

- 回测价格链路改为“多级兜底优先稳定产出”，因为免费数据源存在限流和权限差异，不能把单一源失败等同于整单失败。
- 前端架构改回“单一正式入口优先”，因为当前拆分版存在状态重复和导出链污染，先保证可运行与可维护。
- 执行计划与回测统一按同一持仓集合，是为了解决“推荐5只、回测只算3只”的体验断层。

- yfinance 默认改为 `MARKET_PROXY_MODE=direct`，只对 yfinance 临时禁用代理，其他数据源保持原网络策略。

- 回测和历史价格统一采用“Alpaca -> yfinance -> Alpha -> 6小时缓存”，减少单一数据源失败造成的整单失败。
- 对最终运行失败补了保底错误文案，避免前端出现空白错误提示。
- 对 yfinance 路由增加“有限重试 + 退避”，降低瞬时 429 对单次研究的影响。
- 对实时技术面抓取增加超时后单票降级，避免批量任务超时直接中断整条分析链。
- Yahoo RSS 同步接入“自动路由 + 限速”，并优先保证新闻结果可读性而不是等待长超时。
- 前端逐票卡片新增来源展示（Tech/News/Smart），让用户看到数据来自 Yahoo、Finnhub 或缓存。
- 路由器增加“系统代理自动继承”，是为了避免用户忘填 `MARKET_PROXY_URL` 时 auto 模式失效。
- 批量技术面与 smart money 改为“小批次 + 超时隔离”，是为了避免单只股票卡死拖垮整批。
- 在分析产物中加入 yfinance 路由统计，是为了让失败定位从“猜测”变成“有证据可查”。
- 审计模块优先保证“不崩溃 + 可读返回”，因为该模块失败会直接影响用户对整份报告可信度的判断。
- 当用户明确要求“全局代理”时，优先采用 `.env` 固定配置并清理重复键，避免因旧值优先读取导致行为与预期不一致。
- 同公司多代码默认去重，是为了避免同一发行主体重复占位导致报告与回测结论失真。
- 用户明确点名多个股权类别时保留多个，是为了尊重用户意图并保留可比性。
- 报告卡片改为“技术/新闻分开 + 可点击证据链接”，是为了让非技术用户也能快速验证结论来源。
- 本轮展示策略锁定为“Terminal 优先”，`/debug` 前台隐藏但后端能力保留，是为了把有限时间集中到演示价值最高的部分。

- 错误提示改为“可读解释 + 下一步建议”，是为了减少用户把临时数据源波动误判为系统崩溃。
- 执行计划改为“跟随候选上限”，是为了解决“候选 5 只但执行/回测只有 3 只”的断层。
- 自定义仓位在前端要求总和=100%，是为了避免用户输入异常权重导致结果不可解释。
- 旧回测采用“自动升级覆盖”而非静态兼容，是为了确保单股时间序列和图表能力一致可用。
