# CONTEXT

## 当前正在做什么
- 正在收口终端研究进度卡住的问题，保证本地和云端运行时都能及时显示真实阶段。
- 当前主线是 `main`，本地与 GitHub 已对齐后继续修复。
- 本轮重点是修复数据聚合阶段看起来停在 `58%/59%` 的问题，并把运行中的进度条改成从 `0%` 开始持续增长的数字状态。

## 上次停在哪个位置
- 已完成双报告输出、研究结论页双标签、真实 PDF 导出和四页终端主流程。
- 已修复开发报告字段不一致、风险图缺少数值、结论页拿不到已加载回测图的问题。
- 这轮已把研究进度改成按真实阶段推断，不再长时间停在 `59%`。
- 这轮已把投资 PDF 改成更短的平衡版，只保留原始问题、结论、仓位、风险、证据附录和关键图表。
- 同类细节也一并收口：风险贡献图不再误显示为百分比，开发报告里的“校验提醒”改成更准确的“校验检查”。
- 新发现并修复：匿名用户的研究事件流之前拿不到 `client_id`，页面会停在旧进度，刷新后才看到最终结果。
- 新发现并修复：运行中的 run detail 会被前端缓存命中，事件触发刷新时仍读到旧状态；现在只有已结束 run 才直接用缓存，运行中任务会重新请求后端。
- 已增加兜底轮询：如果浏览器或部署环境漏掉 SSE 事件，queued/running 任务也会每 2.5 秒轻量刷新一次，不需要用户手动刷新页面。
- 已新增成熟工具式首次用户引导：第一次进入终端会看到欢迎弹窗，随后用分步浮层指向开始研究、进度、结论、回测、历史和账户入口；完成或跳过后本地记住，顶部可随时重新打开。
- 正在做 Web 全局交互提速：终端首屏不再预加载历史，历史只在进入历史页时加载；报告完整 memo 改为展开后渲染；回测图表计算已缓存；用户引导去掉平滑滚动和高频测量；终端动效默认降为低负载。
- 新发现并修复：长耗时阶段之前后端只在阶段完成后才写入 step，前端只能推断下一步，所以数据聚合期间会停在 `58%/59%`；现在后端会在数据、证据、报告、校验开始时发布 `running` 阶段，前端会读取真实运行中阶段并把数据聚合显示到 `66%`。
- 新体验调整：运行中重新显示数字百分比，前端从 `0%` 平滑增长到最高 `98%`，任务真正完成后再跳到 `100%`；进度条同时保留流动动效，避免用户误以为任务卡死。

## 近期关键决定与原因
- 账户系统先做本地邮箱密码，不接第三方 OAuth：实现快，适合单服务 Railway 部署。
- 税费只做简化 flat-rate，不做具体地区税务建议。
- 分红只在数据里有现金分红列时纳入，否则明确写入限制说明。
- 自动刷新任务默认关闭，用环境变量开启，避免部署资源不可控。
- `.npm-cache/` 仍是原本存在的未跟踪目录，本轮未处理。
- PDF 导出采用后端统一模板：保留用户原始问题、结论摘要、评分表、逐票卡、风险、完整 memo 和最近回测摘要，避免前端临时 HTML 与页面内容不一致。
- 面试开源项目优先：文档应聚焦产品能力、架构、运行方式和测试结果，不保留临时展示入口。
- Tool Registry 先落地为内部基础层：先统一权限、超时、重试、缓存和审计结果结构，再逐步把现有数据链路接入。
- 双报告来自同一次 run：`report_outputs.investment` 面向投资者，`report_outputs.development` 面向开源项目读者，`final_report` 继续指向投资报告正文以兼容旧逻辑。
- PDF 导出新增 `kind=investment/development`：不传时默认投资报告，开发报告用于说明 Agent、RAG、校验和回测支撑链路。
- 开发报告 diagnostics 统一以 `evidence_count`、`validation_check_count` 为主字段，同时保留旧别名一版兼容历史 run。
- 投资报告图表合同固定为“三类条形图 + 一类回测折线图”：风险图必须带数值，回测图固定使用 `points + summary`。
- 结论页不自动创建回测，但如果当前已经加载过回测，前端会临时覆盖回测图，保证页面、导出和回测页尽量一致。
- 终端进度条改为“阶段加权”而不是“完成步数粗算”：因为 run 只在阶段结束后才写入 step，直接按数量算会在报告生成阶段假卡住。
- EventSource 不能自定义请求头，所以匿名用户的事件流不能再只依赖 `X-Client-Id`：前端改为把 `client_id` 放进事件流 URL，后端同时支持从 query string 读取。
- 投资 PDF 默认采用“平衡版”而不是完整长 memo：PDF 负责可读、可分享，详细长内容继续留在网页、HTML 和 Markdown 导出里。
- PDF 图表统一走“有效图表”逻辑：如果当前 run 已有回测，就优先放组合 vs 基准图；没有回测时，第三张图自动切换成风险贡献图。
- 首页继续保留品牌感和动态背景，但文案必须直接面向用户，不再像项目介绍页。
- 开始研究页采用“三种起点 + 自然语言输入 + 结构化补充 chips + 系统四步预览”组合：用户先判断自己是单次提问、组合研究还是持续跟踪，再进入提问。
- 账户系统沿用现有本地邮箱密码接口，但前台必须给出清晰入口和“同步浏览器记忆”动作，才能让跨设备记忆对用户真正可见。
- 历史页除了回看，还要支持“一键继续跟进”：把旧问题带回提问页，默认转成基于最新数据的连续研究。
- 持续跟踪先做轻量本地列表，不新增复杂后端订阅系统，先验证真实使用感受。
- 用户引导只放在普通终端前台，不进入 `/debug`：普通用户看到功能说明，开发细节继续留在调试页。
- 终端交互优化优先保证“先切页面、再加载数据”：重任务可以继续异步等待，但按钮、页签和报告标签必须先给即时反馈。
- 进度条不能只依赖“已完成步骤数量”：长耗时步骤必须发布 running 事件，否则用户会误以为任务卡死。
- 运行中进度条不应长时间展示固定数字：真实阶段继续保留，数字进度用前端平滑增长表达“系统仍在工作”，完成后才显示 `100%`。
