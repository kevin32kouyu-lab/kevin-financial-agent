# CONTEXT

## 当前正在做什么
- 正在把终端前台从“能用”打磨成“更像成熟金融 agent 产品”的用户体验。
- 当前分支是 `codex/stability-sweep`，基线代码来自最新 `main`。
- 本轮重点是首页精简、开始研究页结构化补充、结论页可信度摘要、账户入口和历史继续跟进。

## 上次停在哪个位置
- 已完成双报告输出、研究结论页双标签、真实 PDF 导出和四页终端主流程。
- 已修复开发报告字段不一致、风险图缺少数值、结论页拿不到已加载回测图的问题。
- 用户前台仍缺少账户感知、持续跟进入口，以及“一眼看懂为什么可信”的摘要层。

## 近期关键决定与原因
- 账户系统先做本地邮箱密码，不接第三方 OAuth：实现快，适合单服务 Railway 部署。
- 税费只做简化 flat-rate，不做具体地区税务建议。
- 分红只在数据里有现金分红列时纳入，否则明确写入限制说明。
- 自动刷新任务默认关闭，用环境变量开启，避免部署资源不可控。
- `.npm-cache/` 仍是原本存在的未跟踪目录，本轮未处理。
- PDF 导出采用后端统一模板：保留用户原始问题、结论摘要、评分表、逐票卡、风险、完整 memo 和最近回测摘要，避免前端临时 HTML 与页面内容不一致。
- 面试开源项目优先：文档应聚焦产品能力、架构、运行方式和测试结果，不保留临时展示入口。
- Tool Registry 先落地为内部基础层：先统一权限、超时、重试、缓存和审计结果结构，再逐步把现有数据链路接入。
- 双报告来自同一次 run：`report_outputs.investment` 面向投资者，`report_outputs.development` 面向开源项目读者，`final_report` 继续指向投资报告正文以兼容旧逻辑。
- PDF 导出新增 `kind=investment/development`：不传时默认投资报告，开发报告用于说明 Agent、RAG、校验和回测支撑链路。
- 开发报告 diagnostics 统一以 `evidence_count`、`validation_check_count` 为主字段，同时保留旧别名一版兼容历史 run。
- 投资报告图表合同固定为“三类条形图 + 一类回测折线图”：风险图必须带数值，回测图固定使用 `points + summary`。
- 结论页不自动创建回测，但如果当前已经加载过回测，前端会临时覆盖回测图，保证页面、导出和回测页尽量一致。
- 首页继续保留品牌感和动态背景，但文案必须直接面向用户，不再像项目介绍页。
- 开始研究页采用“三种起点 + 自然语言输入 + 结构化补充 chips + 系统四步预览”组合：用户先判断自己是单次提问、组合研究还是持续跟踪，再进入提问。
- 账户系统沿用现有本地邮箱密码接口，但前台必须给出清晰入口和“同步浏览器记忆”动作，才能让跨设备记忆对用户真正可见。
- 历史页除了回看，还要支持“一键继续跟进”：把旧问题带回提问页，默认转成基于最新数据的连续研究。
- 持续跟踪先做轻量本地列表，不新增复杂后端订阅系统，先验证真实使用感受。
